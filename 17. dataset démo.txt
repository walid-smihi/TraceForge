## Document de spécifications (spec_checkout.md)

```markdown
# Spécifications Checkout DemoShop v2.3

## Module Stock

### REQ-STOCK-001 : Réservation de stock pendant le paiement
Le système doit réserver l'article dans le stock dès que le client initie le processus
de paiement, afin d'éviter la survente.
La durée de réservation doit être de 15 minutes maximum.
Au-delà de ce délai, l'article est automatiquement remis en stock disponible.

### REQ-STOCK-002 : Disponibilité du stock en temps réel
Le système doit afficher la disponibilité réelle du stock en temps réel sur la fiche produit.
Le délai de mise à jour du stock affiché ne doit pas dépasser 30 secondes.

## Module Paiement

### REQ-PAY-001 : Paiement par carte bancaire
Le système doit accepter les paiements par carte bancaire Visa, Mastercard et American Express.
Le traitement du paiement doit être complété en moins de 10 secondes dans 95% des cas.

### REQ-PAY-002 : Authentification 3D Secure
Le système doit supporter le protocole 3D Secure v2 (3DS2) pour les paiements
nécessitant une authentification forte du client (SCA).
La durée maximale d'attente de la validation bancaire 3D Secure est de 5 minutes.
Si la validation dépasse ce délai, le paiement est annulé et le stock est libéré.

### REQ-PAY-003 : Timeout de session paiement
La session de paiement doit expirer après 20 minutes d'inactivité du client.
Le client doit être notifié 2 minutes avant l'expiration de sa session.

## Sécurité

### REQ-SEC-001 : Non-stockage des données de carte
Les données de carte bancaire (PAN, CVV, date d'expiration) ne doivent jamais être
stockées sur les serveurs DemoShop, même temporairement.
L'intégration doit utiliser une solution de tokenisation certifiée PCI-DSS niveau 1.

### REQ-SEC-002 : Chiffrement des communications
Toutes les communications entre le client, le serveur DemoShop et le prestataire
de paiement doivent utiliser TLS 1.3 minimum.
```

### reservation.service.ts (contenu clé)
```typescript
const RESERVATION_TIMEOUT_MINUTES = 4;  // WARNING: réduit de 15 à 4 en JIRA-421

export class ReservationService {
  async reserveItem(itemId: string, checkoutSessionId: string): Promise<Reservation> {
    const expiresAt = new Date(Date.now() + RESERVATION_TIMEOUT_MINUTES * 60 * 1000);
    return this.db.reservations.create({
      itemId,
      checkoutSessionId,
      expiresAt,
      status: 'active'
    });
  }

  async releaseExpiredReservations(): Promise<void> {
    await this.db.reservations.releaseExpired();
  }
}
```

### three-ds.service.ts (contenu clé)
```typescript
const THREE_DS_MAX_DURATION_MINUTES = 5;

export class ThreeDsService {
  async waitForValidation(paymentIntentId: string): Promise<ValidationResult> {
    const timeoutMs = THREE_DS_MAX_DURATION_MINUTES * 60 * 1000;
    return this.awaitBankConfirmation(paymentIntentId, timeoutMs);
  }
}
```

## Scénario de démo avec ce dataset

Ce dataset permet de démontrer dans l'ordre :
1. L'extraction automatique des 6 exigences depuis spec_checkout.md
2. La détection de REQ-PAY-001 comme ambiguë (pas de percentile sur le 95%)
3. La liaison automatique des exigences aux fichiers de code et tests
4. Le graphe de traçabilité complet (exigences + fichiers + tests + tickets BUG-421)
5. Le conflit critique : reservation=4min vs 3DS max=5min
6. Le rapport d'impact si on modifie REQ-STOCK-001 (timeout 15→4 min)
7. La recommandation : "Ne pas descendre sous 6 minutes, ou suspendre l'expiration pendant la validation bancaire"
