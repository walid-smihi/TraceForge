# Cible Utilisateur — TraceForge

---

## Cible principale V1/V2

Un développeur ou tech lead qui veut comprendre quelles exigences sont liées à quels fichiers, tests et risques.

---

## Personas

### Persona 1 — Le dev solo sur un projet critique

Nom : Lucas, 28 ans, développeur backend freelance
Secteur : fintech, e-commerce, SaaS B2B
Situation : travaille seul ou en petite équipe sur des projets avec des specs rédigées par le client
Douleur principale : les specs évoluent souvent, et il n'a aucune visibilité sur ce qui change vraiment et ce que ça impacte dans son code
Ce qu'il veut : un outil qui lui dit "si tu changes ça, tu dois aussi regarder là"
Ce qu'il ne veut pas : un outil lourd qui lui demande de tout saisir manuellement

### Persona 2 — Le tech lead en équipe agile

Nom : Amira, 34 ans, tech lead dans une ESN
Secteur : industrie, transport, finance
Situation : manage une équipe de 5 développeurs, reçoit des specs en Word/PDF, doit s'assurer que tout est bien couvert par les tests
Douleur principale : elle passe du temps à faire des revues manuelles pour vérifier que les exigences critiques ont bien des tests associés
Ce qu'elle veut : un dashboard de couverture et une alerte automatique quand une exigence critique n'a pas de test
Ce qu'elle ne veut pas : un outil SaaS qui envoie le code et les specs en dehors de l'entreprise

### Persona 3 — Le développeur qui prépare une démo portfolio

Nom : Walid, 29 ans, développeur full-stack
Secteur : cherche un poste dans une entreprise type Thales, Capgemini, Safran
Situation : construit un projet personnel ambitieux pour montrer ses compétences techniques et produit
Douleur principale : veut montrer qu'il maîtrise les concepts d'architecture, de qualité logicielle et d'IA appliquée
Ce qu'il veut : un projet impressionnant, bien structuré, avec une vraie démo narrative
Ce qu'il ne veut pas : un projet CRUD banal sans valeur ajoutée

---

## Non-cibles pour la V1

- Les équipes de plus de 10 personnes (pas de multi-user en V1)
- Les entreprises qui veulent héberger en SaaS cloud (V5)
- Les clients qui ont besoin de connecteurs JIRA/Confluence complets (V3+)
- Les non-développeurs (product managers, business analysts) — ils ne savent pas lire les fichiers de code
- Les projets sans code source accessible (ex : projets 100% no-code)

---

## Contextes d'utilisation

Contexte 1 — Projet avec cahier des charges PDF
L'utilisateur reçoit un PDF de 50 pages de specs. Il uploade le fichier, TraceForge extrait les exigences, et il peut les relier à son code existant.

Contexte 2 — Refactoring risqué
L'utilisateur veut modifier une règle métier sensible. Il utilise TraceForge pour voir tous les fichiers liés avant de toucher au code.

Contexte 3 — Revue de sprint
Le tech lead utilise TraceForge avant une sprint review pour vérifier que toutes les exigences du sprint ont bien un code et un test associés.

Contexte 4 — Démo portfolio
Le développeur utilise le dataset DemoShop pour montrer les fonctionnalités de l'app (extraction, graphe, conflit timeout, rapport d'impact).

---

## Ce qui déclenche l'adoption

- L'utilisateur a vécu un bug de prod causé par un changement d'exigence mal compris
- L'utilisateur est fatigué de faire des revues manuelles "est-ce que cette spec est couverte par un test ?"
- L'utilisateur veut impressionner un recruteur avec un projet qui n'est pas un simple CRUD
