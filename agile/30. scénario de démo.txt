# Scénario de démo — TraceForge

Durée cible : 5-10 minutes
Public cible : recruteur technique, tech lead, développeur senior

---

## Pré-requis

L'app est démarrée et le projet DemoShop est préchargé (make seed-demo).
Les résultats sont pré-calculés : pas besoin de lancer Ollama pendant la démo.

---

## Pitch d'ouverture (30 secondes)

"Je vais vous montrer TraceForge, un outil que j'ai construit pour résoudre un problème réel : quand une exigence change dans un projet, personne ne sait exactement ce que ça impacte dans le code.

Pour la démo, j'utilise un projet e-commerce fictif — DemoShop — avec de vraies spécifications PDF, un vrai repo de code, et un bug qui s'est passé en production il y a quelques jours."

---

## Étape 1 — Dashboard et overview (1 minute)

Action : ouvrir http://traceforge.dev → dashboard → cliquer sur DemoShop

Ce que vous montrez :
- "Voici le projet DemoShop. L'app a déjà analysé les spécifications et le code."
- Métriques : 6 exigences | 8 fichiers | 9 liens | 1 conflit critique
- "Ce badge rouge, c'est ce qu'on va expliquer."

Point technique à souligner :
- "L'extraction a été faite via LLM — ici Ollama local, les données n'ont jamais quitté la machine."

---

## Étape 2 — Exigences extraites (1.5 minutes)

Action : aller dans l'onglet Exigences

Ce que vous montrez :
- "L'app a extrait 6 exigences depuis le PDF de spécifications."
- Montrer REQ-STOCK-001 : "Réservation de stock pendant le paiement — 15 minutes."
- Ouvrir REQ-PAY-001 → badge orange "Ambiguë" : "95% des cas — l'app a détecté qu'il manque une métrique pour les 5% restants."
- Montrer le type (fonctionnelle, sécurité, performance) et la source (section du document d'origine)

Point technique :
- "Chaque exigence est corrigeable manuellement. L'IA suggère, l'humain valide."

---

## Étape 3 — Graphe de traçabilité (2 minutes)

Action : aller dans l'onglet Graphe

Ce que vous montrez :
- "Voici le graphe qui relie tout : exigences, fichiers de code, tests, et tickets JIRA."
- Identifier les nœuds bleus (exigences), verts (fichiers), orange (tests), violets (tickets)
- Cliquer sur BUG-421 (violet) : "Ce bug JIRA est directement lié à reservation.service.ts et three-ds.service.ts"
- Activer le filtre "exigences sans test" : "Aucune exigence sans test ici, tout est couvert."
- Montrer l'arête rouge pointillée entre REQ-STOCK-001 et REQ-PAY-002 : "Et voilà le conflit."

Point technique :
- "Les liens ont été suggérés par similarité sémantique vectorielle (pgvector), puis validés. Les traits pointillés = liens non encore validés."

---

## Étape 4 — Le conflit critique (2 minutes)

Action : aller dans l'onglet Impact → sélectionner REQ-STOCK-001

Ce que vous montrez :
- "Ce bug en production vient d'ici. Un développeur a réduit le timeout de réservation de 15 minutes à 4 minutes pour optimiser le stock."
- Montrer la section "Conflits détectés" : "REQ-STOCK-001 définit 4 minutes. REQ-PAY-002 définit que la validation 3D Secure peut prendre jusqu'à 5 minutes."
- "L'app a automatiquement détecté que la réservation expire avant la fin possible du paiement bancaire."

Section "Fichiers impactés" :
- "reservation.service.ts est directement lié."
- "payment.service.ts est lié indirectement, à distance 2 dans le graphe."

Section "Tests à rejouer" :
- "reservation-timeout.test.ts et three-ds-long-flow.test.ts — exactement les deux tests qui auraient détecté le bug."

Point technique :
- "L'analyse d'impact fait un BFS de profondeur 2 dans le graphe pour trouver les fichiers indirects."

---

## Étape 5 — Rapport d'impact (1 minute)

Action : cliquer "Exporter en Markdown"

Ce que vous montrez :
- Le rapport s'ouvre / se télécharge
- "Voici ce qu'on peut envoyer à un tech lead avant de merger une PR."
- Sections : Résumé | Risques | Fichiers concernés | Tests à rejouer | Recommandation

Recommandation générée :
"Ne pas descendre sous 6 minutes de réservation, ou suspendre l'expiration pendant la validation bancaire 3D Secure."

---

## Étape 6 — Paramètres IA (30 secondes, optionnel)

Action : Paramètres IA

Ce que vous montrez :
- "L'app tourne par défaut avec Ollama local — aucune donnée envoyée à l'extérieur."
- "Si on veut de meilleures performances, on peut brancher OpenAI avec sa propre clé."
- Tracking tokens : "On voit exactement ce que chaque opération a consommé."

---

## Clôture (30 secondes)

"En résumé, TraceForge fait trois choses :
1. Il extrait les exigences d'un document de spécifications.
2. Il les relie automatiquement au code et aux tests via des embeddings vectoriels.
3. Il détecte les incohérences et génère un rapport d'impact avant que le bug arrive en production.

Le stack : FastAPI, Next.js, PostgreSQL + pgvector, Ollama, Docker Compose. Tout tourne en local."

---

## Points forts à souligner si questions techniques

Architecture :
- Tâches lourdes (scan repo, extraction, embeddings) en asynchrone via Redis Queue
- Embeddings stockés dans PostgreSQL via pgvector (pas de base vectorielle séparée)
- Interface LLMProvider abstraite → changer de modèle sans toucher au code métier
- StorageService abstrait → migrer vers S3 sans refactoring

Qualité :
- Tests unitaires et d'intégration avec pytest, MockProvider pour la CI
- CI GitHub Actions : lint + typecheck + build + tests à chaque PR
- Definition of Done stricte : loading/error/empty states obligatoires
- ADRs documentant toutes les décisions architecturales importantes

Extensibilité :
- Multi-user prévu en V2 (table User + JWT, ADR-005)
- Connecteurs Git (commits, PRs) prévus en V3
- SaaS multi-tenant prévu en V5

## FAQ recruteurs

Q : "Pourquoi pas LangChain ?"
R : "Pour garder le contrôle sur les prompts et éviter une dépendance externe pour des opérations simples. J'ai une interface LLMProvider custom qui fait exactement ce dont j'ai besoin."

Q : "Comment tu gères les hallucinations du LLM ?"
R : "Les suggestions sont toujours soumises à validation humaine. Le JSON est parsé avec Pydantic avec retry si le format est invalide. L'utilisateur corrige ce qui est faux."

Q : "Ça marche avec n'importe quel projet ?"
R : "En V1, avec des docs PDF/Markdown et des repos en Python, TypeScript, JavaScript. Le parser de code (tree-sitter) est extensible à d'autres langages."
