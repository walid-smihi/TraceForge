# User Stories — TraceForge MVP

Format : En tant que [rôle], je veux [action], afin de [bénéfice].
Critères d'acceptation au format bullet list.

---

## US-001 — Créer un projet

En tant que développeur,
je veux créer un nouveau projet avec un nom et un domaine métier,
afin d'organiser mon travail de traçabilité par projet.

Critères d'acceptation :
- Je peux saisir un nom (obligatoire), une description (optionnel) et choisir un domaine (liste : e-commerce, fintech, santé, industrie, autre)
- Le projet apparaît immédiatement dans mon tableau de bord après création
- Je ne peux pas créer deux projets avec le même nom
- L'app affiche une erreur claire si le nom est vide

---

## US-002 — Importer un document de spécifications

En tant que développeur,
je veux importer un fichier PDF ou Markdown de spécifications,
afin d'extraire les exigences sans les recopier à la main.

Critères d'acceptation :
- Je peux uploader un fichier .pdf, .md, .txt ou .docx par glisser-déposer ou sélection
- L'app affiche une barre de progression pendant le traitement
- L'app affiche les sections détectées après le traitement
- L'app affiche une erreur claire si le fichier dépasse 50 MB ou a un format non supporté
- Je peux uploader plusieurs documents dans le même projet

---

## US-003 — Consulter les exigences extraites

En tant que développeur,
je veux voir la liste des exigences extraites de mon document,
afin de vérifier que l'extraction est correcte avant de continuer.

Critères d'acceptation :
- Je vois un tableau avec : ID (REQ-001), titre, type, priorité, statut
- Je peux filtrer par type (fonctionnelle, sécurité, performance, etc.)
- Je peux rechercher par mot-clé dans le titre ou la description
- Les exigences ambiguës sont signalées visuellement (badge orange)
- Je peux cliquer sur une exigence pour voir son détail complet

---

## US-004 — Corriger une exigence extraite

En tant que développeur,
je veux corriger le titre ou la description d'une exigence mal extraite,
afin d'avoir des exigences précises avant de les relier au code.

Critères d'acceptation :
- Je peux modifier le titre, la description, le type et la priorité d'une exigence
- Les modifications sont sauvegardées immédiatement
- Si l'exigence a des liens validés, ils passent automatiquement en statut "à revoir"
- Je peux créer une exigence manuellement si elle n'a pas été extraite

---

## US-005 — Importer un dépôt de code

En tant que développeur,
je veux importer mon dépôt de code local,
afin que l'app puisse analyser les fichiers et proposer des liens avec les exigences.

Critères d'acceptation :
- Je peux saisir un chemin absolu vers un dossier local ou uploader un .zip
- Je peux fournir une URL GitHub publique
- L'app affiche une barre de progression pendant le scan
- L'app exclut automatiquement : node_modules, .git, dist, build, __pycache__
- Je reçois un compte rendu : X fichiers scannés, Y fichiers de tests détectés, Z fichiers ignorés

---

## US-006 — Consulter les fichiers analysés

En tant que développeur,
je veux voir la liste des fichiers analysés avec leur résumé,
afin de vérifier que l'app a bien compris le rôle de chaque fichier.

Critères d'acceptation :
- Je vois un tableau avec : chemin, langage, rôle détecté, nombre de liens, est un test (oui/non)
- Je peux filtrer par langage et par "sans lien" ou "tests uniquement"
- Je peux cliquer sur un fichier pour voir son résumé complet et ses liens

---

## US-007 — Valider des liens suggérés

En tant que développeur,
je veux valider ou rejeter les liens suggérés entre exigences et fichiers,
afin de construire une carte de traçabilité fiable.

Critères d'acceptation :
- L'app m'affiche les suggestions triées par score de confiance (plus haut en premier)
- Chaque suggestion montre : l'exigence, le fichier, le score, la justification
- Je peux cliquer "Valider", "Rejeter" ou "Ignorer" pour chaque suggestion
- Les liens validés apparaissent en trait plein dans le graphe
- Je peux créer un lien manuel depuis la fiche d'une exigence

---

## US-008 — Visualiser le graphe de traçabilité

En tant que tech lead,
je veux voir un graphe interactif qui relie exigences, fichiers et tests,
afin d'avoir une vue globale de la couverture de mon projet.

Critères d'acceptation :
- Le graphe affiche des nœuds colorés par type (exigence=bleu, fichier=vert, test=orange)
- Je peux zoomer, déplacer et réorganiser le graphe
- Je peux filtrer pour n'afficher que les "exigences sans test"
- Je peux cliquer sur un nœud pour voir ses détails sans quitter le graphe
- Le graphe se met à jour en temps réel quand je valide un lien

---

## US-009 — Voir les exigences sans couverture

En tant que tech lead,
je veux voir quelles exigences n'ont aucun test associé,
afin d'identifier les zones risquées du projet avant une livraison.

Critères d'acceptation :
- Un filtre "sans test" dans la liste des exigences
- Un badge de compte sur le tableau de bord du projet
- Dans le graphe, les exigences sans test sont visuellement distinctes (rouge ou icône d'alerte)

---

## US-010 — Générer un rapport d'impact

En tant que développeur,
je veux savoir quels fichiers et tests sont impactés par une modification d'exigence,
afin de réduire les risques de régression avant de coder le changement.

Critères d'acceptation :
- Je peux déclencher une analyse d'impact depuis la fiche d'une exigence
- L'app liste les fichiers directement liés et les fichiers indirectement liés (à distance 2 dans le graphe)
- L'app liste les tests qui valident ces fichiers
- L'app signale les conflits potentiels détectés
- Je peux exporter le rapport en Markdown

---

## US-011 — Détecter un conflit de timeout

En tant que développeur,
je veux que l'app détecte automatiquement un conflit entre deux exigences avec des durées incompatibles,
afin d'éviter un bug de prod avant qu'il n'arrive.

Critères d'acceptation :
- L'app détecte deux exigences avec des valeurs numériques temporelles dans le même domaine
- Elle affiche une alerte claire : "Risque : X peut expirer avant la fin de Y"
- L'alerte apparaît dans la liste des conflits du projet et dans le rapport d'impact
- Je peux marquer un conflit comme "faux positif" ou "résolu"

---

## US-012 — Configurer le provider IA

En tant que développeur,
je veux configurer le provider LLM utilisé par l'app (Ollama, OpenAI, ou Mock),
afin de contrôler le coût et la confidentialité des analyses.

Critères d'acceptation :
- Depuis les paramètres, je peux choisir : Mock, Ollama, OpenAI, Mistral
- Pour Ollama : je peux saisir l'URL de mon instance et le modèle (ex : llama3.1:8b)
- Pour OpenAI/Mistral : je peux saisir ma clé API (masquée, jamais affichée en clair)
- Je peux tester la connexion avant de sauvegarder
- L'app affiche les tokens consommés et le coût estimé pour la session en cours
