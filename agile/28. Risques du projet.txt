# Risques du projet — TraceForge

Format : Risque | Probabilité | Impact | Mitigation

---

## Risques techniques

### RISK-T01 — Extraction d'exigences peu fiable

Probabilité : haute (LLM n'est pas déterministe)
Impact : élevé (fonctionnalité cœur du produit)

Mitigation :
- Permettre la correction manuelle de chaque exigence extraite
- Afficher les sections du document source pour vérification
- Tester sur plusieurs formats (PDF natif, Markdown, DOCX)
- Utiliser un prompt structuré avec schéma JSON forcé
- MockProvider avec résultats prédéfinis pour les tests auto

Indicateur de succès : > 80% des exigences extraites correctement sur le dataset DemoShop

---

### RISK-T02 — Suggestions de liens exigences-code peu pertinentes

Probabilité : moyenne (embeddings seuls peuvent être insuffisants)
Impact : élevé (fonctionnalité différenciante)

Mitigation :
- Combiner embeddings vectoriels ET analyse LLM pour les candidats proches
- Permettre la validation/rejet manuel de toutes les suggestions
- Afficher la justification de chaque suggestion
- Threshold configurable (min_score)
- Commencer avec des liens manuels si les suggestions sont mauvaises

Indicateur de succès : > 70% des liens DemoShop suggérés correctement dans le top-3

---

### RISK-T03 — Performance dégradée sur gros repos

Probabilité : moyenne
Impact : moyen (démo sur des petits repos, problème réel en V3+)

Mitigation :
- Tronquer les fichiers > 5000 lignes
- Limiter le scan aux extensions pertinentes
- Générer les embeddings en batch
- Afficher la progression pour les longs traitements
- Mettre en cache les résumés et embeddings (ne pas recalculer si fichier inchangé)

Seuil acceptable : scan d'un repo de 200 fichiers en < 10 min avec Ollama

---

### RISK-T04 — Coût LLM trop élevé

Probabilité : faible (OpenAI gpt-4o-mini est très peu cher)
Impact : faible (mode Ollama local = $0)

Mitigation :
- Mode Ollama local par défaut ($0)
- Tracker les tokens et coûts dans llm_usage
- Afficher le coût estimé avant les analyses importantes
- gpt-4o-mini recommandé pour le cloud (très bon rapport qualité/prix)

---

### RISK-T05 — Graphe illisible sur de gros projets

Probabilité : moyenne (beaucoup de nœuds = graphe confus)
Impact : moyen (écran graphe central pour la démo)

Mitigation :
- Filtres par type de nœud (exigences uniquement, tests uniquement, etc.)
- Filtre "exigences sans test" pour focus sur les zones risquées
- Clustering auto si > 50 nœuds (React Flow le supporte)
- Export PNG pour partager une vue fixe

---

### RISK-T06 — LLM retourne du JSON invalide

Probabilité : moyenne (surtout avec llama3.1:8b)
Impact : moyen (job échoue)

Mitigation :
- Retry automatique avec prompt de correction si JSON invalide (1 retry)
- Validation stricte avec Pydantic après parsing
- Fallback sur un résultat partiel si possible
- Logs détaillés de la réponse brute en cas d'échec

---

## Risques produit

### RISK-P01 — Produit trop complexe à comprendre

Probabilité : moyenne
Impact : élevé (bloque l'adoption même par un recruteur)

Mitigation :
- Projet DemoShop préchargé avec résultats déjà calculés
- Onboarding : first-run experience qui guide vers la démo
- Scénario de démo narratif (doc + vidéo)
- Interface claire : 1 action principale par écran

---

### RISK-P02 — Pas de valeur claire en démo

Probabilité : faible si démo bien préparée
Impact : élevé (objectif principal du projet)

Mitigation :
- Démo centrée sur le cas DemoShop (conflit timeout = bug réel de prod)
- Rapport d'impact = artefact tangible exportable
- Préparer une vidéo démo de 3-5 minutes avec narration
- Avoir une instance prod déployée accessible en ligne

---

### RISK-P03 — Trop de features, pas assez de polish

Probabilité : haute (risque classique de sur-engineering)
Impact : élevé (démo bancale = mauvaise impression)

Mitigation :
- Sprint 7 entièrement dédié au polish et à la démo
- Definition of Done stricte (loading/error/empty states obligatoires)
- MVP scope limité et explicite (Hors MVP documenté)
- Règle : une feature à 80% vaut mieux que deux features à 60%

---

## Risques planning

### RISK-PL01 — Scope trop grand

Probabilité : haute
Impact : élevé

Mitigation :
- Roadmap versionnée V0→V6 : on sait où s'arrêter
- Hors MVP explicitement documenté dans le fichier 3
- Revue du scope en début de chaque sprint

---

### RISK-PL02 — Sous-estimation du frontend

Probabilité : haute (le frontend prend toujours plus de temps que prévu)
Impact : moyen

Mitigation :
- Utiliser Shadcn/ui (composants prêts à l'emploi)
- Sprint planning avec 30% de buffer frontend
- Maquettes détaillées (fichier 7) pour éviter les allers-retours design

---

### RISK-PL03 — Envie de tout faire enterprise dès le début

Probabilité : haute (syndrome du développeur ambitieux)
Impact : élevé (ne jamais finir la V1)

Mitigation :
- Hors MVP documenté et visible
- Sprint 0 = setup uniquement, pas de features
- ADR-005 : no auth in V1 est une décision assumée
