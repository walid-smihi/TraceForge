# V0 — Prototype jetable

## Objectif

Prouver que l’idée marche sur un mini-exemple.

Tu ne fais pas encore une vraie app complète. Tu fais une preuve de concept.

## Fonctionnalités

- Upload d’un fichier Markdown ou texte.
- Extraction simple des exigences.
- Affichage en tableau.
- Ajout manuel de liens vers des fichiers de code.
- Détection basique des mots-clés :
  - paiement,
  - timeout,
  - stock,
  - sécurité,
  - utilisateur,
  - test.

## Exemple

Tu uploades :

```txt
Le système doit réserver l’article pendant 15 minutes.
Le système doit supporter un paiement 3D Secure pouvant durer 5 minutes.
```

L’app affiche :

```txt
REQ-001 — Réserver l’article pendant 15 minutes
REQ-002 — Supporter le paiement 3D Secure pendant 5 minutes
```

Et détecte :

```txt
Contrainte temporelle détectée :
reservation_timeout = 15 min
payment_duration = 5 min
```

## Tech possible

- FastAPI
- SQLite
- simple frontend Next.js ou même Streamlit
- pas forcément de LLM au début

## Durée

1 à 2 semaines.

## But

Te rassurer que le concept est faisable.

# V1 — MVP local utilisable

## Objectif

Avoir une première vraie app locale que tu peux montrer.

L’utilisateur peut créer un projet, uploader des documents, importer un repo et voir les exigences.

## Fonctionnalités

### Projet

- Créer un projet.
- Ajouter un nom, une description, un domaine métier.

### Documents

- Upload :
  - PDF texte,
  - Markdown,
  - TXT,
  - DOCX éventuellement.
- Extraction du texte.
- Découpage en sections.

### Exigences

- Extraction des exigences.
- Attribution automatique d’un ID :

```txt
REQ-001
REQ-002
REQ-003
```

- Type d’exigence :
  - fonctionnelle,
  - sécurité,
  - performance,
  - disponibilité,
  - conformité,
  - interface.

### Interface

- Tableau des exigences.
- Fiche détail pour chaque exigence.
- Recherche simple.
- Tags métier.

### Liens manuels

L’utilisateur peut relier manuellement :

```txt
REQ-001 → src/payment/checkout.ts
REQ-002 → tests/payment/three-ds.test.ts
```

## Ce que la V1 ne fait pas encore

- Pas encore d’analyse intelligente profonde.
- Pas encore de Jira.
- Pas encore de vrai graphe.
- Pas encore de SaaS.
- Pas encore de multi-utilisateur.

## Tech recommandée

```txt
Frontend : Next.js + TypeScript + Tailwind
Backend : FastAPI
DB : PostgreSQL
Parsing : PyMuPDF, python-docx
Auth : simple login local
```

## Durée

4 à 6 semaines.

## Ce que tu peux montrer

> “Voici une app qui transforme un cahier des charges en exigences structurées et permet de les relier au code.”

Déjà propre.

# V2 — MVP impressionnant portfolio

## Objectif

Passer de “outil simple” à “wow, il y a une vraie intelligence produit”.

C’est probablement la meilleure version pour attirer des recruteurs.

## Fonctionnalités ajoutées

### Import repo GitHub/local

- Upload ou connexion à un repo.
- Scan des fichiers :
  - `.ts`,
  - `.js`,
  - `.py`,
  - `.java`,
  - `.md`,
  - fichiers de tests.

### Résumé des fichiers

Pour chaque fichier :

```txt
src/checkout/reservation.service.ts
Rôle détecté : gestion de la réservation d’articles pendant le paiement.
Entités : article, stock, checkout, expiration.
Constantes détectées : RESERVATION_TIMEOUT_MINUTES = 15.
```

### Recherche sémantique

Tu peux chercher :

```txt
réservation pendant paiement
```

Et l’app trouve aussi :

```txt
checkout hold
stock lock
reservation timeout
cart expiration
```

Même si les mots exacts sont différents.

### Suggestions de liens

L’app propose :

```txt
REQ-001 semble liée à :
- src/checkout/reservation.service.ts — confiance 91%
- tests/checkout/reservation-timeout.test.ts — confiance 84%
```

L’utilisateur peut :

```txt
Valider
Refuser
Modifier
```

### Graphe interactif

Vue type :

```txt
REQ-001
  → checkout.service.ts
  → reservation.service.ts
  → reservation-timeout.test.ts
  → BUG-014
```

Avec React Flow.

### Analyse de qualité des exigences

L’app détecte :

```txt
"Le système doit être rapide."
```

Et dit :

```txt
Exigence floue.
Problème : aucun seuil mesurable.
Suggestion : préciser une latence maximale.
```

### Première analyse d’impact

Si une exigence change, l’app affiche :

```txt
Exigence modifiée :
Durée de réservation : 15 min → 4 min

Impact probable :
- src/checkout/reservation.service.ts
- tests/payment/three-ds-long-flow.test.ts
- REQ-PAY-003
```

### Détection simple de conflits

Exemple :

```txt
reservation_timeout = 4 minutes
three_ds_max_duration = 5 minutes
```

Alerte :

```txt
Risque : la réservation expire avant la fin possible du paiement 3D Secure.
```

## Tech ajoutée

```txt
pgvector
embeddings locaux ou API
React Flow
Tree-sitter pour analyser le code
Ollama ou OpenAI/Mistral optionnel
```

## Durée

8 à 12 semaines.

## Ce que tu peux montrer

Là, tu as une vraie démo portfolio forte :

> “J’upload une spec et un repo, l’app extrait les exigences, trouve les fichiers liés, affiche un graphe et détecte des risques métier indirects.”

C’est probablement la première version vraiment impressionnante.

# V3 — Beta produit pour vrais projets

## Objectif

Rendre l’app utilisable sur un vrai projet open source ou un vrai projet perso.

Ici, on sort du simple portfolio.

## Fonctionnalités ajoutées

### Intégration Git complète

- Analyse des commits.
- Historique des modifications.
- Détection des fichiers changés.
- Association entre commits et exigences.

Exemple :

```txt
Commit a83f9d2 :
"reduce reservation timeout from 15 to 4 minutes"

Impact détecté :
REQ-STOCK-001
REQ-PAY-003
TEST-THREEDS-002
```

### Pull request analysis

L’utilisateur connecte une PR GitHub/GitLab.

L’app dit :

```txt
Cette PR modifie un timeout critique.
Exigences impactées :
- REQ-STOCK-001
- REQ-PAY-003

Tests recommandés :
- three-ds-long-validation.test.ts
- reservation-expiration.test.ts
```

### Import tickets

Au début, tu peux commencer simple :

- CSV export Jira,
- CSV GitHub Issues,
- JSON.

Puis plus tard API directe.

### Bugs liés

L’app relie :

```txt
BUG-421 : clients perdent leur article pendant 3D Secure
```

à :

```txt
REQ-STOCK-001
REQ-PAY-003
reservation.service.ts
three-ds.service.ts
```

### Moteur de règles

Tu ajoutes des règles métier configurables.

Exemple :

```txt
reservation_timeout >= external_payment_duration + safety_margin
```

Ou :

```txt
session_timeout >= max_checkout_duration
```

Ou :

```txt
token_expiration >= max_user_validation_duration
```

### Rapport d’impact

Export PDF/Markdown :

```txt
Analyse d’impact — modification REQ-STOCK-001

Résumé :
La durée de réservation est passée de 15 minutes à 4 minutes.

Risque :
Incompatibilité avec la durée maximale 3D Secure de 5 minutes.

Fichiers concernés :
...

Tests à rejouer :
...

Recommandation :
Ne pas descendre sous 6 minutes ou suspendre l’expiration pendant la validation bancaire.
```

## Durée

3 à 5 mois depuis le début.

## Niveau atteint

À partir de là, ton projet devient vraiment sérieux.

Tu peux faire une vidéo LinkedIn/GitHub qui donne envie à un recruteur technique de te contacter.

# V4 — Version Pro utilisable en équipe

Passer de “outil solo” à “outil d’équipe”.

Là, tu commences à penser comme un vrai produit.

## Fonctionnalités ajoutées

### Multi-utilisateur

- Organisation.
- Projet.
- Membres.
- Rôles :
  - admin,
  - engineer,
  - reviewer,
  - viewer.

### Workflow de validation

Quand l’app suggère un lien :

```txt
REQ-001 → reservation.service.ts
```

Un utilisateur peut :

```txt
Proposer
Valider
Rejeter
Commenter
```

Exemple :

```txt
Lien validé par Alice le 15/06/2026.
```

Tout est tracé :

```txt

Pourquoi ?
```

### Dashboard qualité

Exemples :

```txt
Exigences sans code : 8

Exigences ambiguës : 23

Risques critiques : 3
```

### Mode local-first

L’app peut tourner :

```txt
100% local
Docker Compose
sans envoyer les données dehors
```

### Providers IA configurables

```txt
Ollama local
OpenAI
Mistral
Anthropic
Azure OpenAI
Custom endpoint
```

### Suivi des coûts LLM

Dashboard :

```txt

```

## Durée

5 à 8 mois depuis le début.

# V5 — Version SaaS commercialisable

## Objectif

Pouvoir vendre l’app à des freelances, petites équipes, startups, ESN ou équipes produit.

Pas encore forcément aux grands groupes défense/aéro, mais déjà à de vrais clients.

## Fonctionnalités nécessaires

### SaaS multi-tenant

Chaque entreprise a son espace isolé :

```txt
Organisation A
Organisation B
Organisation C
```

Les données ne se mélangent jamais.

### Paiement

- Stripe.
- Plans :
  - Free,
  - Pro,
  - Team,
  - Business.

Exemple :

```txt
Free : 1 projet, 50 exigences
Pro : 10 projets, 1000 exigences
Team : multi-utilisateur, intégrations Git
Business : SSO, support, gros quotas
```

### Quotas

Limiter :

```txt
nombre de projets
nombre de documents
taille de repo
nombre d’analyses IA
budget LLM mensuel
```

### Onboarding

Un utilisateur doit comprendre l’app sans toi.

Donc il faut :

- projet démo préchargé,
- tutoriel,
- exemples,
- documentation,
- tooltips,
- vidéo de démo.

### Sécurité minimale sérieuse

- Auth solide.
- Gestion des permissions.
- Chiffrement des secrets API.
- Backups.
- Rate limiting.
- Logs.
- Monitoring.
- Suppression de compte/données.

### Intégrations

Minimum commercial :

- GitHub.
- GitLab.
- Jira ou Linear.
- Export CSV/PDF.
- Webhooks.

### Landing page

Une vraie page commerciale :

```txt
TraceForge helps engineering teams connect requirements, code, tests and risks.
```

Avec :

- vidéo,
- screenshots,
- pricing,
- FAQ sécurité,
- documentation,
- contact.

## Durée

8 à 12 mois depuis le début si tu bosses sérieusement.

## Niveau atteint

À partir de là, tu peux essayer d’avoir des premiers utilisateurs payants.
