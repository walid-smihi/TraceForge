# Stratégie Git — TraceForge

---

## Branches principales

main    → version stable, toujours déployable
dev     → intégration, branche de travail principale

Règle : on ne pousse jamais directement sur main. On merge dev → main via PR quand une version est stable.

---

## Branches de fonctionnalité

Nommage : feature/{sprint}-{description-courte}

Exemples :
  feature/s1-project-crud
  feature/s1-document-upload
  feature/s2-requirement-extraction
  feature/s3-repo-scan
  feature/s4-trace-links-suggestions

Branches de correction :
  fix/{description}
  fix/pdf-extraction-error
  fix/embedding-null-pointer

Branches de refactoring :
  refactor/{description}
  refactor/split-document-parser

---

## Flux de travail

1. Créer la branche depuis dev :
   git checkout dev && git pull origin dev
   git checkout -b feature/s1-project-crud

2. Travailler en petits commits (Conventional Commits)

3. Push la branche :
   git push origin feature/s1-project-crud

4. Ouvrir une PR vers dev

5. Vérifier que la CI passe (lint + tests)

6. Merger la PR dans dev (squash or merge commit)

7. Supprimer la branche après merge

---

## Conventional Commits

Format : <type>(<scope>): <description>

Types autorisés :
  feat     → nouvelle fonctionnalité
  fix      → correction de bug
  refactor → refactoring sans changement de comportement
  test     → ajout ou modification de tests
  docs     → documentation
  chore    → tâches de maintenance (deps, config, etc.)
  perf     → amélioration de performance
  ci       → modification des workflows CI

Exemples :
  feat(requirements): add automatic type classification
  fix(pdf): handle empty pages in PyMuPDF extraction
  refactor(llm): split OllamaProvider into text and embed clients
  test(trace-links): add fixtures for confidence score edge cases
  docs(api): update endpoint documentation for /jobs
  chore(deps): bump FastAPI to 0.115
  ci: add codecov upload step

Règles :
- Description en anglais
- Imperative mood : "add" et non "added" ou "adding"
- Pas de point final
- Scope optionnel mais recommandé

---

## Tags de version

Schéma : v{MAJOR}.{MINOR}.{PATCH}

Exemples :
  v0.1.0 → fin du Sprint 1 (projets + documents)
  v0.2.0 → fin du Sprint 2 (exigences)
  v0.5.0 → V1 locale complète (Sprints 1-4)
  v1.0.0 → V2 portfolio (graphe + IA + impact)

Créer un tag :
  git tag -a v0.1.0 -m "Sprint 1 complete: project CRUD + document upload"
  git push origin v0.1.0

## Règles de protection des branches (GitHub)

main :
- Require PR before merging
- Require status checks to pass (CI)
- No direct pushes

dev :
- Require CI to pass before merge (recommandé dès le Sprint 2)

## Pull Request template (.github/pull_request_template.md)

```markdown
## Description
<!-- Quoi et pourquoi -->

## Changements
- [ ] feat / fix / refactor / test / docs / chore

## Tests
- [ ] Tests unitaires ajoutés / mis à jour
- [ ] Tests API ajoutés / mis à jour
- [ ] Testé manuellement dans Docker

## Checklist
- [ ] La CI passe
- [ ] Pas de clé secrète exposée
- [ ] Le .env.example est à jour si nouvelle variable
- [ ] La documentation est mise à jour si nécessaire
```
