# Règles Métier — TraceForge

Ces règles sont implémentées dans conflict_detection_service.py et trace_link_service.py.

---

## Règles de qualité des exigences

RULE-001 — Exigence critique sans test
Condition : une exigence avec priority="critical" n'a aucun TraceLink validé vers un CodeFile de type test.
Sévérité : critical
Action : créer un DetectedConflict, badge rouge dans l'UI.
Implémentation : vérifier lors de la validation d'un lien et en tâche de fond quotidienne.

RULE-002 — Conflit de valeurs numériques temporelles
Condition : deux exigences du même projet contiennent des constantes numériques temporelles (min, sec, heures) qui pourraient être incompatibles (ex : timeout A < durée maximale B).
Sévérité : critical
Action : créer un DetectedConflict avec description explicite.
Implémentation : regex pour extraire les valeurs numériques + unités, comparaison par domaine métier.
Exemple : "réservation = 4 min" vs "3D Secure max = 5 min" → conflit.

RULE-003 — Lien à revoir après modification d'exigence
Condition : une exigence passe en status="modified" (son contenu a changé).
Action : tous les TraceLinks avec status="validated" liés à cette exigence passent en status="needs_review".
Implémentation : hook SQLAlchemy after_update sur Requirement.

RULE-004 — Exigence ambiguë sans métrique
Condition : le titre ou la description d'une exigence contient un des mots : "rapide", "simple", "robuste", "performant", "scalable", "fiable", "stable", "efficace", "léger" sans valeur numérique associée.
Action : marquer is_ambiguous=True avec ambiguity_reason généré.
Sévérité : warning
Implémentation : analyse lexicale au moment de la sauvegarde de l'exigence.

RULE-005 — Fichier récemment modifié sans exigence liée
Condition : un CodeFile a un content_hash différent du dernier scan et aucun TraceLink validé.
Sévérité : info
Action : badge "non tracé" dans la liste des fichiers.
Implémentation : détecter lors d'un rescan du repository.

RULE-006 — Exigence de sécurité sans test de sécurité
Condition : une exigence avec req_type="security" n'a aucun TraceLink validé vers un fichier de test.
Sévérité : critical
Action : créer un DetectedConflict.
Implémentation : même logique que RULE-001, filtré sur req_type="security".

RULE-007 — Exigence de performance sans seuil numérique
Condition : une exigence avec req_type="performance" ne contient aucune valeur numérique (ms, %, RPS, req/s).
Sévérité : warning
Action : marquer is_ambiguous=True, ambiguity_reason="Exigence de performance sans seuil mesurable".
Implémentation : regex sur description + titre au moment de la sauvegarde.

RULE-008 — Score de confiance trop faible pour validation automatique
Condition : un TraceLink suggéré a confidence_score < 50.
Action : le lien ne peut pas être validé automatiquement, il reste en status="suggested" jusqu'à validation manuelle.
Implémentation : vérification dans trace_link_service avant tout changement de status automatique.

RULE-009 — Lien manuel toujours validé
Condition : un TraceLink est créé manuellement (is_manual=True).
Action : confidence_score = 100.00, status = "validated" immédiatement.
Implémentation : dans la route POST /trace-links avec is_manual=True.

RULE-010 — Alerte couverture insuffisante
Condition : plus de 50% des exigences d'un projet n'ont aucun TraceLink validé.
Sévérité : warning
Action : bannière d'avertissement sur le dashboard du projet.
Implémentation : calcul dans /api/v1/projects/{id}/metrics.

RULE-011 — Exigence dépréciée non modifiable
Condition : une exigence a status="deprecated".
Action : interdire la création de nouveaux TraceLinks vers cette exigence.
Implémentation : validation dans trace_link_service avant création.

RULE-012 — Conflit de durée de session
Condition : une exigence contient un timeout de session et une autre exigence contient une durée de checkout ou de paiement supérieure.
Sévérité : critical
Exemple : "session expire après 20 min" vs "checkout peut prendre 25 min".
Implémentation : cas spécial de RULE-002 sur les domaines session/auth.

RULE-013 — Fichier de test sans exigence liée
Condition : un CodeFile est_test_file=True et n'a aucun TraceLink validé vers une exigence.
Sévérité : info
Action : badge "test orphelin" dans la liste des fichiers.
Implémentation : lors du calcul des métriques.

RULE-014 — Cohérence des identifiants d'exigences
Condition : deux exigences du même projet ont le même code (REQ-XXX).
Action : refuser la création, retourner une erreur 409.
Implémentation : contrainte UNIQUE (project_id, code) en base + gestion d'erreur.

RULE-015 — Propagation de l'impact à distance 2
Condition : lors d'une analyse d'impact, les fichiers directement liés à l'exigence modifiée sont identifiés.
Action : les fichiers liés à ces fichiers (distance 2 dans le graphe) sont aussi inclus dans le rapport d'impact.
Implémentation : parcours BFS à profondeur 2 dans impact_service.py.
